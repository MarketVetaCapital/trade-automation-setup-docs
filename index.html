Overview

This meeting covered the complete manual deployment process for the Discord WebSocket Forwarding Service (Trade Listener) on AWS ECS. The following topics were covered:
  • Pushing a Dockerized app from local → AWS ECR
  • Configuring secrets via AWS Systems Manager Parameter Store
  • Setting up ECS Task Definitions and Clusters
  • Networking and Security Groups for port 8765
  • Troubleshooting IAM permissions
  • CloudWatch logs for debugging
  • Updating environment variables for Webhook routing
1

AWS ECS Setup

1

Authenticate & Prepare Local Environment

  • Ensure Docker and AWS CLI are installed on your local machine
  • If credential errors occur: IAM → Users → Security Credentials → Create Access Key
  • Use the new Access Key ID and Secret Access Key to authenticate AWS CLI
2

Push Docker Image to ECR

  • AWS Console → Search Elastic Container Registry (ECR)
  • Select repository (e.g., discord-self) → Click "View push commands"
  • Select your OS (macOS/Linux or Windows)
  • Open terminal → Navigate to project directory
  • Run the provided commands one by one: login, build, tag, push
Terminal Commands
cd self   # Navigate to project directory

# Then run ECR push commands from AWS Console
# (login → build → tag → push)
aws ecr get-login-password --region us-east-2 | docker login ...
docker build -t discord-self .
docker tag discord-self:latest <ECR_URI>:latest
docker push <ECR_URI>:latest
⚠️

Windows Users Use PowerShell — NOT standard CMD — to run ECR deployment commands.

3

Create ECS Cluster

  • AWS Console → Elastic Container Service (ECS) → Clusters → Create cluster
  • Provide a cluster name
  • Use low-spec configuration — estimated cost: $20–$30/month
2

Services Configuration

1

Configure Secrets in Parameter Store

  • Navigate: Systems Manager → Parameter Store → Create parameter
  • Enter parameter name (e.g., Discord Token)
  • Set Type: SecureString
  • Paste your Discord token → Click Save
2

Create Task Definition

  • ECS → Task Definitions → Create new task definition
  • Name: discord-listener or trade-listener
  • Resources: 1 CPU, standard memory
  • Task execution role: Create default role
  • Container image: Paste Image URI from ECR
  • Port mapping: 8765
  • Add environment variables (see table below)
Variable Name Value Notes
DISCORD_TOKEN From Parameter Store SecureString reference
SOURCE_CHANNEL_ID Discord Channel ID Channel to monitor
DS_ENABLED true Enable Discord service
HOST Host value
DESTINATION_TYPE Destination type
WEBHOOK_URL Webhook endpoint Trade Executor webhook
3

Create ECS Service & Security Group

  • Go to Cluster → Services tab → Create
  • Name: discord-ser
  • Networking → Create new Security Group → Name: socket
  • Add rule: Custom TCP → Port 8765 → Source: Anywhere
  • Click Create
🔴

Security Warning Port 8765 is currently open to public internet (testing only). Must be restricted to internal IPs before production.

4

Updating Service with New Variables

  • Task Definitions → select definition → Create new revision
  • Update environment variable values → Click Create
  • Clusters → Service → Update service
  • Select new task definition revision number → Click Update
⚠️

Dynamic IP Every service update assigns a NEW public IP. Update your WebSocket client with the new IP after each update.

Azure Entra ID Configuration

1

Problem: Redirect to localhost after login

  • After Microsoft login in production, users were redirected to localhost instead of the Cloudflare Pages URL
  • Azure App Registration only had localhost redirect URIs configured

Problem: AADSTS9002326 — Cross-origin token redemption

  • Login failed with error AADSTS9002326: cross-origin token redemption is permitted only for SPA redirect URIs
  • Azure app was registered as type "Web", which does not support the SPA auth flow used by MSAL in the browser

Solution: Changed the Azure App Authentication platform type from Web to Single-Page Application (SPA), and ensured the production URL is listed under SPA redirect URIs.

🔴

Required for MSAL The app registration must use the SPA platform — not Web — or Microsoft login will fail with AADSTS9002326 in production.

Overview

The meeting covered the end-to-end setup for a new Discord listener application designed to forward messages from a Discord server to mobile phones via SMS. Key points:
  • Built in Java using the JDA (Java Discord API) library, replacing an older Python-based approach
  • Uses internal WebSockets to listen for incoming Discord messages in real time — no traditional web server required
  • On message receipt, triggers an API call to the Vital Global SMS service to broadcast to specific users
  • Packaged as a .jar file and deployed to the cloud — does not rely on a local machine
ℹ️

Architecture shift While the previous Python listener used AWS ECS, this Java application is deployed on AWS Elastic Beanstalk.

Services Configuration

1

Discord Bot Configuration

  • Enable Developer Mode in Discord: Settings → Advanced → Developer Mode
  • Open the Discord Developer Portal (discord.com/developers/applications)
  • Create a new application/bot (e.g., named My Bot)
  • Go to Bot settings → click Reset Token to generate a new secure connection token
  • Enable intents: check Server Members Intent and Message Content Intent so the bot can read server messages
  • Connect the bot to your server: Server Settings → Integrations → App → link My Bot
2

Java Application (Listener)

  • Integrate the JDA (Java Discord API) library into the project
  • Configure authentication using the generated Discord Bot token
  • The application uses WebSockets internally to instantly detect when a message is received
  • On message receipt, forward content via the Vital Global API (see below)
3

Vital Global (SMS Alerts) Configuration

  • The application calls the Vital Global API to forward received Discord messages as SMS
  • Use a GET request, passing configuration as query parameters
  • Sender: The From address must be the Vital Global registered admin mobile number
  • Recipients: Provide a comma-separated list of target phone numbers for simultaneous delivery to multiple users
API pattern (example)
GET https://<vital-global-api>?from=<admin-number>&to=<num1,num2,num3>&message=<encoded-text>
# Actual parameters per Vital Global API documentation

Overview

This meeting covered the end-to-end manual deployment of a Discord WebSocket forwarding service (also referred to as a listener bot or trade listener) to AWS Elastic Container Service (ECS). Topics covered:
  • Pushing a local Docker container to AWS Elastic Container Registry (ECR)
  • Setting up AWS Systems Manager Parameter Store for secure credentials
  • Configuring an ECS Task Definition and Cluster
  • Exposing the service via an ECS Service and Security Group
  • Troubleshooting via CloudWatch logs
  • Testing the WebSocket connection using Postman
  • Estimated monthly costs of around $20–$30 (low-spec configuration)
1

AWS ECS Setup

1

Generate AWS CLI Credentials (If Needed)

If your local environment is not authenticated to push to AWS, you must generate access keys:

  • Navigate to IAM via the AWS Console search bar
  • Go to Users and select your IAM user
  • Open the Security credentials tab
  • Click Create access key to get an Access Key ID and Secret Access Key
  • Configure these in your local terminal to authenticate the AWS CLI
IAM
→
Users
→
Security credentials
→

2

Push Docker Image to Elastic Container Registry (ECR)

  • Navigate to Elastic Container Registry in the AWS Console
  • Click on your repository (e.g., discord-self)
  • Click View push commands in the top right
  • Select your operating system (macOS/Linux or Windows)
  • Open your terminal — use PowerShell on Windows, not standard CMD
  • Navigate to your project directory (e.g., cd self)
  • Copy and paste the provided commands one by one: login, build, tag, and push to ECR
Terminal Commands
cd self   # Navigate to project directory

# ECR push commands from AWS Console → View push commands
aws ecr get-login-password --region us-east-2 | \
  docker login --username AWS --password-stdin <ECR_URI>

docker build -t discord-self .
docker tag discord-self:latest <ECR_URI>:latest
docker push <ECR_URI>:latest
⚠️

Windows Users Run deployment commands in PowerShell — NOT standard Command Prompt (cmd).

3

Create an ECS Cluster

  • Navigate to Elastic Container Service (ECS)
  • On the left sidebar, select Clusters
  • Click Create cluster
  • Provide a cluster name and proceed with creation
  • Use low-spec capacity to keep costs low (~$20–$30/month)
2

Services Configuration

1

Configure Secrets in Parameter Store

  • Search for Systems Manager in the AWS Console
  • On the left sidebar, click Parameter Store
  • Click Create parameter
  • Enter the parameter name (e.g., Discord token)
  • Change the Type to SecureString
  • Paste your Discord token into the value field and click Save
2

Create the Task Definition

  • Navigate to ECS → Task definitions
  • Click Create new task definition
  • Name the definition (e.g., discord-listener or trade-listener)
  • Keep resources at the lowest available spec (e.g., 1 CPU, low RAM)
  • Under Task execution role, select Create default role
  • In Container details, paste the Image URI from ECR
  • Under Port mappings, map the container port to 8765
  • Scroll to Environment variables and add required configurations
Variable Name Value Notes
DISCORD_TOKEN From Parameter Store SecureString reference
SOURCE_CHANNEL_ID Discord Channel ID Channel to monitor
DS_ENABLED true Enable Discord service
HOST Host value
DESTINATION_TYPE Destination type
WEBHOOK_URL Webhook endpoint Missing value causes "No web hook URL configured" error
3

Create the ECS Service and Networking

  • Go to Clusters and click into your newly created cluster
  • Under the Services tab, click Create
  • Name the service (e.g., discord-ser)
  • Scroll to the Networking section
  • Select Create a new security group and name it socket
  • Under security group rules: Add rules → Custom TCP → port 8765 → source Anywhere (public internet)
  • Click Create to launch the service
🔴

Security Vulnerability Port 8765 is open to the public internet for testing. Anyone who finds the IP can intercept WebSocket messages. Restrict to internal IPs before production.

Methods Discussed in the Meeting

1

Manual Deployment Method

Push code directly from a local development machine to AWS ECR using AWS CLI and Docker CLI commands, bypassing GitHub for the initial setup.

2

Environment Variable Update Method

To change configurations (webhook URL, target accounts, etc.) on a running ECS service:

  • Task Definitions → Create new revision
  • Add or edit environment variables → Save
  • Cluster → Service → Update service → select the new revision number
  • Wait for restart — container receives a new public IP
3

Log Debugging Method

When a service fails to route messages, navigate to the specific ECS Task → Logs, or open CloudWatch Logs to find trace errors — missing tokens, missing webhooks, permission failures, etc.

4

WebSocket Verification Method

Use Postman's WebSocket interface connected to the ECS Task's public IP on port 8765 to verify Discord messages are successfully ingested and forwarded by the container.
